# 后端 Go 代码风格规范

## 核心原则

后端代码要像一份清晰的服务结构图。

阅读者打开文件后，应该马上看明白：

- 这个文件属于哪个后端模块。
- 这个模块负责什么业务能力。
- 对外暴露了哪些接口或 Wails 方法。
- 依赖了哪些服务、配置、上下文或文件资源。
- 主要业务流程从哪里开始，到哪里结束。
- 错误在哪里产生、在哪里包装、在哪里返回。
- 哪些函数是当前模块专属，哪些函数是共用工具。

优先保证职责清楚、边界清楚、调用链清楚。不要把入口、业务、工具函数、错误处理混在一起。

## 文件开头注释

每个 Go 后端源码文件开头都必须有中文注释框，说明文件职责。

除非用户明确要求，或者文件职责真实变化，否则不要改动文件级注释。

使用这个格式：

```go
/*
|--------------------------------------------------------------------------
| 文件模块名称
|--------------------------------------------------------------------------
| 用中文说明这个 Go 文件主要负责什么，属于哪个后端能力或服务流程。
|--------------------------------------------------------------------------
*/
```

## 文件内部推荐顺序

Go 文件建议按这个顺序组织：

1. 文件开头注释。
2. `package`。
3. `import`。
4. 模块能力清单。
5. 常量和配置。
6. 核心结构体和依赖字段。
7. 构造函数。
8. 生命周期方法。
9. 对外接口或 Wails 绑定方法。
10. 业务流程函数。
11. 模块专属辅助函数。
12. 共用工具函数放到底部。

这个顺序的目标是：先看模块轮廓，再看对外能力，最后看实现细节。

## 模块能力清单

在 `import` 后面，用中文注释框列出当前文件负责的后端能力。

示例：

```go
/*
|--------------------------------------------------------------------------
| 模块能力清单
|--------------------------------------------------------------------------
| 文件导入：接收前端传入的模型文件路径。
| 格式识别：判断模型文件属于 STL、OBJ、FBX、STEP 或 SolidWorks。
| 状态返回：向前端返回当前文件是否可直接预览或需要服务端转换。
|--------------------------------------------------------------------------
*/
```

能力清单只写模块级职责，不写逐行语法解释。

## 结构体和依赖

结构体前必须说明它代表什么后端角色。

结构体字段要按依赖类型分组：

- 运行时上下文。
- 配置。
- 外部服务。
- 缓存或状态。
- 文件系统或路径。

示例：

```go
/*
|--------------------------------------------------------------------------
| 模型转换服务
|--------------------------------------------------------------------------
| 负责管理模型格式识别、转换任务创建，以及转换状态查询。
|--------------------------------------------------------------------------
*/
type ModelTranscodeService struct {
	ctx      context.Context
	workDir  string
	cacheDir string
}
```

## 构造函数

构造函数要靠近结构体，并说明初始化了哪些依赖。

示例：

```go
/*
|--------------------------------------------------------------------------
| 模型转换服务创建
|--------------------------------------------------------------------------
| 初始化转换服务需要的工作目录和缓存目录。
|--------------------------------------------------------------------------
*/
func NewModelTranscodeService(workDir string, cacheDir string) *ModelTranscodeService {
	return &ModelTranscodeService{
		workDir:  workDir,
		cacheDir: cacheDir,
	}
}
```

## 对外接口和 Wails 方法

前端会调用的方法必须集中放在“对外接口”区域。

每个对外方法前都要说明：

- 前端什么时候调用。
- 输入参数代表什么。
- 返回值代表什么。
- 是否只是查询，还是会触发任务。

示例：

```go
/*
|--------------------------------------------------------------------------
| 查询模型预览状态
|--------------------------------------------------------------------------
| 前端选中文件后调用。返回该格式能否直接预览，或是否需要服务端转换。
|--------------------------------------------------------------------------
*/
func (a *App) PreviewStatus(path string) (PreviewStatus, error) {
	if path == "" {
		return PreviewStatus{}, ErrEmptyModelPath
	}

	return a.modelService.GetPreviewStatus(path)
}
```

## 业务流程函数

业务流程函数要按执行顺序组织，不要把主流程拆得过碎。

推荐结构：

1. 参数校验。
2. 读取或准备输入。
3. 执行业务动作。
4. 保存结果或状态。
5. 返回明确结果。

示例：

```go
/*
|--------------------------------------------------------------------------
| 创建模型转换任务
|--------------------------------------------------------------------------
| 校验输入文件，生成转换任务，并返回任务编号给前端查询进度。
|--------------------------------------------------------------------------
*/
func (s *ModelTranscodeService) CreateTask(inputPath string, targetFormat string) (TaskResult, error) {
	if inputPath == "" {
		return TaskResult{}, ErrEmptyModelPath
	}

	if targetFormat == "" {
		return TaskResult{}, ErrEmptyTargetFormat
	}

	task := s.buildTask(inputPath, targetFormat)

	return task, nil
}
```

## API 和函数归属规则

如果某个函数只服务一个对外接口，就放在该接口下方附近。

如果某个函数会被多个接口复用，就放到底部“共用工具函数”区域。

不要把共用工具插在一个业务接口和它的专属流程之间。

## 错误处理规范

错误要做到“读路径时能看明白失败原因”。

要求：

- 可预期错误定义成清晰命名的变量或类型。
- 不吞掉错误。
- 返回给前端的错误要可读，不要只返回底层库的模糊报错。
- 包装错误时说明当前业务动作。

示例：

```go
var ErrEmptyModelPath = errors.New("模型文件路径不能为空")

func wrapModelLoadError(path string, err error) error {
	return fmt.Errorf("读取模型文件失败 %s: %w", path, err)
}
```

## 代码对齐和可读性

后端代码也要避免横向挤成一长串。

要求：

- 结构体字段按语义分组，必要时对齐。
- 配置对象多字段时竖向展开。
- 返回结构体时多字段竖向展开。
- 长参数列表换行。
- 函数名使用明确业务含义，不用模糊缩写。

推荐：

```go
return PreviewStatus{
	FileName:        fileName,
	SourceFormat:    sourceFormat,
	CanPreview:      canPreview,
	NeedConversion:  needConversion,
	PreviewFormat:   previewFormat,
}
```

避免：

```go
return PreviewStatus{FileName: fileName, SourceFormat: sourceFormat, CanPreview: canPreview, NeedConversion: needConversion, PreviewFormat: previewFormat}
```

## 服务端目录拆分建议

当功能变多时，优先按业务能力拆目录，而不是按“工具函数大杂烩”拆目录。

推荐方向：

- `model_import`：模型导入、格式识别、文件校验。
- `model_preview`：预览格式判断、预览产物管理。
- `model_convert`：格式转换任务、转换队列、转换结果。
- `runtime_bridge`：Wails 和前端通信入口。
- `storage`：缓存目录、临时文件、导出文件。

每个目录内也要保持文件职责单一。

## 验证要求

修改 Go、Wails 或 server 后端代码后：

1. 运行 `gofmt`。
2. 在 `backend_go` 目录运行 `go build ./...`。
3. 如果影响前端绑定或前端产物，运行前端构建。
4. 如果当前目录是 git 仓库，按项目要求处理 pull 和 push。
